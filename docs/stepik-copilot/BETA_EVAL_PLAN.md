# Beta Eval Plan

## Цель

Проверить, что Stepik Copilot уже дает полезную учебную поддержку на реальных
шагах Stepik, не уходит в готовые ответы и не требует преждевременной сложной
инфраструктуры вроде auth, billing или глубокой Stepik API-интеграции.

Это промежуточный gate между Phase 2 и Phase 3.

## Что проверяем

- полезность ответа для понимания текущего шага;
- качество подсказок на тестовых и кодовых заданиях;
- устойчивость guardrails против прямого ответа;
- работу на шагах с комментариями и без комментариев;
- понятность пустых, error и loading-состояний;
- задержку ответа на выбранном provider;
- готовность локального feedback loop к ручному разбору кейсов.

## Eval Set

Минимальный набор: 20-30 реальных шагов Stepik.

Состав:

- 5-8 theory/text шагов;
- 5-8 choice шагов;
- 5-8 code шагов;
- 2-4 video/unknown шагов;
- минимум 8 шагов с содержательными комментариями;
- минимум 5 шагов без видимых комментариев;
- минимум 5 коротких или неоднозначных шагов, где модель должна честно
  признать нехватку контекста.

Для каждого шага фиксируем:

- URL;
- тип шага;
- есть ли комментарии;
- provider и модель;
- время ответа;
- выбранный режим: explain, hint или notes;
- итоговую оценку ответа.

## Rubric

Каждый ответ получает одну основную оценку:

- `useful` — помогает понять шаг, увидеть риск или проверить себя;
- `too_direct` — слишком близко подводит к готовому ответу;
- `missed_context` — пропускает важный контекст шага или комментариев;
- `factual_error` — содержит неверный вывод или искажает материал.

Дополнительно отмечаем:

- `latency_seconds`;
- `has_direct_answer_leak`;
- `needs_prompt_change`;
- `needs_extraction_fix`;
- `notes`.

## Pass Criteria

Переход к Phase 3 разрешен, если:

- `useful` не ниже 70%;
- `has_direct_answer_leak` равен 0 для choice/code шагов;
- `factual_error` не выше 5%;
- p95 latency близко к 8 секундам или есть понятное решение по provider;
- для шагов без комментариев ответ корректно деградирует;
- error state не показывает пользователю внутренние детали реализации;
- команда вручную разобрала минимум 10 негативных кейсов.

Если критерии не выполнены, Phase 3 откладывается. Следующая итерация должна
быть prompt/extraction/UX pass, а не auth или monetization.

## Процесс

1. Собрать eval set в таблицу или issue.
2. Прогнать каждый шаг через установленное расширение.
3. Сохранить оценку через feedback-кнопки.
4. Для негативных оценок зафиксировать краткую причину.
5. Сгруппировать проблемы:
   - prompt;
   - extraction;
   - provider latency/quality;
   - UI clarity;
   - guardrails.
6. Выбрать не больше трех fixes на следующую итерацию.
7. Повторить eval только на затронутых категориях.

## Не делаем в этой фазе

- не добавляем обязательную авторизацию;
- не строим billing;
- не подключаем Stepik API ради полноты контекста;
- не добавляем историю анализов в облако;
- не расширяем продукт на другие LMS.

Все это имеет смысл только после подтверждения, что текущий core loop полезен.

# Architecture

## Архитектурный принцип

Начать с простой схемы:

- расширение читает то, что уже видит пользователь на странице;
- серверная часть отвечает за AI-анализ;
- Stepik API подключается позже, если он действительно нужен для контекста,
  истории или более надежного доступа к структуре курса.

## Компоненты

### 1. Browser Extension

Состав:

- `content script` — читает DOM текущей страницы Stepik;
- `sidebar UI` — показывает ответ агента и управляет состояниями;
- `background/service worker` — управляет запросами, хранением токена и
  событиями расширения;
- `storage` — локальные настройки пользователя.

Ответственность:

- определить, что пользователь находится на релевантной странице;
- извлечь содержимое шага;
- собрать комментарии и метаданные;
- отправить данные в backend;
- отрисовать структурированный ответ.

### 2. Backend API

Основные ручки:

- `POST /analyze-step`
- `POST /summarize-comments`
- `GET /health`

Ответственность:

- принимать нормализованный payload от расширения;
- валидировать входные данные;
- собирать prompt для модели;
- применять guardrails;
- возвращать структурированный JSON.

### 3. LLM Layer

Задачи:

- summary текущего шага;
- выделение pitfalls;
- извлечение сигналов из комментариев;
- формирование короткого checklist;
- подавление прямых ответов и спойлеров.

### 4. Optional Data Layer

Появляется, если понадобятся:

- история анализов;
- пользовательские лимиты;
- подписка;
- персональные настройки;
- сравнение шагов между сессиями.

## Предлагаемый стек

### Extension

- `Plasmo` или чистый `Chrome Extension Manifest V3`
- `React`
- `TypeScript`

### Backend

- `FastAPI` или `Node.js`
- `Pydantic` или `Zod` для схем
- `Redis` опционально для rate limiting и кэша

### Storage

- `PostgreSQL` для пользователей и истории
- `localStorage` / `chrome.storage` для локальных настроек

## Data Flow

1. Пользователь открывает шаг Stepik.
2. Content script собирает:
   - URL;
   - название курса/урока/шага;
   - текст текущего шага;
   - текст комментариев;
   - дополнительные DOM-сигналы, если они нужны.
3. Sidebar отправляет нормализованный payload в backend.
4. Backend формирует prompt и вызывает модель.
5. Модель возвращает JSON:
   - summary;
   - pitfalls;
   - comment insights;
   - checklist;
   - warnings.
6. Sidebar рендерит ответ пользователю.

## Authentication Strategy

### Вариант A: без обязательной auth на старте

Подходит для локального MVP и раннего тестирования.

Плюсы:

- минимальный порог входа;
- быстрее собрать первую рабочую версию;
- меньше интеграционного риска.

Минусы:

- сложнее привязать подписку и историю;
- нет кросс-девайсного опыта;
- меньше контроля над usage.

### Вариант B: auth в собственном сервисе

Подходит для production-версии.

Плюсы:

- платные планы;
- история;
- лимиты;
- персонализация.

Минусы:

- выше сложность;
- больше surface area по privacy.

### Вариант C: глубокая интеграция со Stepik API

Нужна только при наличии реальной ценности от:

- загрузки соседних шагов;
- доступа к структуре курса не через DOM;
- более точной привязки к user progress.

## Recommended Technical Sequence

1. DOM-only extraction.
2. Backend with one analysis endpoint.
3. Structured UI in sidebar.
4. Prompt hardening.
5. Optional own auth.
6. Optional Stepik API integration.
